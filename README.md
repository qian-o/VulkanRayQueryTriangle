# Vulkan descriptor-heap acceleration-structure investigation

Minimal headless C++20 case for loading a
`RaytracingAccelerationStructure` through `VK_EXT_descriptor_heap` and using it
in an inline ray query.

The shader is generated by Slang 2026.13 from:

```hlsl
DescriptorHandle<RaytracingAccelerationStructure> Scene;
RaytracingAccelerationStructure scene = constants.Scene;
```

The host follows the Vulkan descriptor-heap API path: it queries the
acceleration-structure descriptor size with
`vkGetPhysicalDeviceDescriptorSizeEXT`, writes a
`VK_DESCRIPTOR_TYPE_ACCELERATION_STRUCTURE_KHR` descriptor with
`vkWriteResourceDescriptorsEXT`, places it after the implementation-reserved
heap range with the required alignment, and binds the resource heap before a
`1x1x1` compute dispatch.

## Current finding

Slang lowers the descriptor-heap access to an integer load followed by an
acceleration-structure conversion:

```text
OpDecorate %_runtimearr_ulong ArrayStride 8
OpDecorate %slang_resourceHeap BuiltIn ResourceHeapEXT
%38 = OpUntypedAccessChainKHR ... %_runtimearr_ulong ...
%40 = OpLoad %ulong %38
%42 = OpConvertUToAccelerationStructureKHR %41 %40
```

This does not match the published Vulkan descriptor-heap type correspondence:

- `SPV_EXT_descriptor_heap` defines a descriptor as an opaque object with a
  descriptor type such as `OpTypeAccelerationStructureKHR`; it does not list
  an integer type as an acceleration-structure descriptor type.
- The Descriptor Heap Interface says resources retrieved from a heap must have
  descriptors matching the declared resource type.
- The Heap Resource Type Correspondence table maps
  `VK_DESCRIPTOR_TYPE_ACCELERATION_STRUCTURE_KHR` in `ResourceHeapEXT` to
  `OpTypeAccelerationStructureKHR`.
- A typed heap read consumes the implementation-reported size of that
  descriptor. Slang instead fixes the array stride and load width at 8 bytes.
- `VUID-RuntimeSpirv-Result-11350` also describes an
  `OpLoad` whose result type is `OpTypeAccelerationStructureKHR` when its
  pointer is derived from `ResourceHeapEXT`.

Vulkan does allow a 64-bit acceleration-structure address stored as ordinary
data to be consumed by `OpConvertUToAccelerationStructureKHR`; in that case no
descriptor is required. That is distinct from this host path, which places an
API-generated, opaque acceleration-structure descriptor in the heap. No
published rule was found that permits those descriptor bytes to be
reinterpreted as the raw address supplied to the writer.

The ordinary-address alternative is described in the
[`VK_EXT_mutable_descriptor_type` proposal](https://github.com/KhronosGroup/Vulkan-Docs/blob/main/proposals/VK_EXT_mutable_descriptor_type.adoc).

`spirv-val` accepts the module, but standalone SPIR-V validation cannot know
whether the bytes loaded at runtime are ordinary address data or an opaque
Vulkan descriptor.

Relevant specification sections:

- https://docs.vulkan.org/spec/latest/chapters/interfaces.html#interfaces-resources-descriptorheap
- https://docs.vulkan.org/spec/latest/chapters/interfaces.html#interfaces-resources-heap-type-correspondence
- https://docs.vulkan.org/spec/latest/appendices/spirvenv.html#VUID-RuntimeSpirv-Result-11350
- https://github.khronos.org/SPIRV-Registry/extensions/EXT/SPV_EXT_descriptor_heap.html

The descriptor payload printed by the host is diagnostic only. Vulkan treats
the generated descriptor representation as opaque; equality with a
`VkDeviceAddress` is not an application-visible guarantee. The tested NVIDIA
implementation reports an 8-byte acceleration-structure descriptor, but that
does not make an 8-byte integer representation portable.

## Build and validate

Requirements:

- CMake 3.24 or newer and a C++20 compiler
- Vulkan SDK 1.4 with `VK_EXT_descriptor_heap` revision 1 headers
- A device supporting the extensions and features checked by `main.cpp`
- `VK_LAYER_KHRONOS_validation` for `--validate-only`

```powershell
cmake -S . -B build
cmake --build build --config Release
spirv-val --target-env vulkan1.4 .\ray_query_heap.spv
.\build\Release\vulkan_descriptor_heap_ray_query_repro.exe --validate-only
```

`--validate-only` enables core and synchronization validation and records the
AS builds, heap binding, dispatch, and barriers without submitting work. On
the tested RTX 3050 Laptop GPU with NVIDIA 610.74 it completes without a
validation message and reports `queue submissions=0`.

The standard descriptor API revision has not been used as evidence of a
driver failure. A full dispatch should not be used for vendor attribution
until the shader lowering is confirmed to satisfy the Vulkan type rules.

If Slang instead defines `DescriptorHandle<RaytracingAccelerationStructure>`
as indexing ordinary 64-bit address data rather than API-generated
descriptors, that contract must be documented and the host must store the raw
address rather than call `vkWriteResourceDescriptorsEXT`. The current Slang
documentation and lowering do not make this distinction clear.

## Regenerate the shader

Use Slang 2026.13 explicitly; the Vulkan SDK installation may contain a
different `slangc` version.

```powershell
$slangc = "C:\path\to\Slang\2026.13\bin\slangc.exe"
& $slangc -version
& $slangc .\ray_query_heap.slang `
  -target spirv -profile sm_6_6 -entry CSMain -stage compute `
  -capability spvDescriptorHeapEXT -capability spirv_1_6 `
  -fvk-use-entrypoint-name -o .\ray_query_heap.spv
spirv-dis .\ray_query_heap.spv -o .\ray_query_heap.spvasm
```

Checked-in artifact hashes:

```text

ray_query_heap.spvasm:  20650B871BA8F8E1942C755CA52BEAA0157599770561764A3FF2EC9F9B2271AA
```

## Reporting

Open the current issue first in the Slang tracker:

- https://github.com/shader-slang/slang/issues
- Draft: [ISSUE_DRAFT.md](ISSUE_DRAFT.md)

The existing Slang issues and fixes establish that the current lowering is
intentional, but they do not address Vulkan's heap resource type
correspondence:

- https://github.com/shader-slang/slang/issues/10671
- https://github.com/shader-slang/slang/pull/11209
- https://github.com/shader-slang/slang/issues/11231
- https://github.com/shader-slang/slang/pull/11494

If Slang maintainers confirm that the integer-load sequence is intended to be
valid Vulkan, request clarification from both relevant specification owners and
cross-link the issues:

- https://github.com/KhronosGroup/SPIRV-Headers/issues
- https://github.com/KhronosGroup/Vulkan-Docs/issues

`SPV_EXT_descriptor_heap` explicitly directs extension questions to the
SPIRV-Headers tracker. Use the NVIDIA bug portal only after the lowering is
confirmed valid and the same committed standard-API revision reproduces an
actual dispatch failure:

- https://developer.nvidia.com/nvidia_bug/add

Warning: running the executable without `--validate-only` submits the workload
and may reset or hang the GPU and desktop.
